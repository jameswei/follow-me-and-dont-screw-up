# 验证工作流 (Validation Workflow)
# 位置: shared/validation/validation-workflow.md
# 用途: 定义代码实现后的验证闭环流程

# =============================================================================
# 核心原则: 验证即代码，自动化优先
# =============================================================================

## 验证金字塔

```
         ┌─────────┐
         │  E2E    │  ← 用户场景验证 (少量)
         │  Tests  │
        ┌┴─────────┴┐
        │ Integration│ ← 组件交互验证 (中量)
        │   Tests   │
       ┌┴───────────┴┐
       │    Unit      │ ← 函数/类验证 (大量)
       │    Tests     │
      ┌┴──────────────┴┐
      │  Static Analysis│ ← 类型检查、Lint、格式化
      │   (编译/检查)   │
     ┌┴─────────────────┴┐
     │   Syntax/Compile   │ ← 基础语法/编译检查
     │      Check         │
     └────────────────────┘
```

## 验证闭环流程

### 每次代码变更后的强制验证顺序

```
┌─────────────────────────────────────────────────────────────┐
│ Step 1: 语法/编译检查                                         │
│ 目标: 代码能解析/编译                                          │
│ 工具: 编译器、解释器、语法检查器                                 │
│ 失败处理: 立即停止，修复语法错误                                 │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 2: 静态分析                                              │
│ 目标: 类型正确、符合规范、无常见错误                             │
│ 工具: TypeScript/Lint/SAST                                     │
│ 失败处理: 修复所有 Error，评估 Warning                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 3: 单元测试                                              │
│ 目标: 函数/类行为正确                                          │
│ 工具: Jest/pytest/Go test                                      │
│ 失败处理: 修复失败的测试，补充缺失的测试                          │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 4: 集成测试 (如适用)                                      │
│ 目标: 组件间交互正确                                           │
│ 工具: 测试数据库、API 测试、容器                                 │
│ 失败处理: 修复集成问题，检查接口契约                              │
└─────────────────────────────────────────────────────────────┘
                              ↓
┌─────────────────────────────────────────────────────────────┐
│ Step 5: 代码审查 (Self-Review)                                │
│ 目标: 符合规范、无 obvious 问题                                 │
│ 工具: 代码 diff、规范检查清单                                    │
│ 失败处理: 重构、补充文档、优化实现                                │
└─────────────────────────────────────────────────────────────┘
```

## 失败处理策略

### 失败分类与响应

| 失败类型 | 严重程度 | Agent 响应 | 用户通知 |
|---------|---------|-----------|---------|
| 编译错误 | 🔴 阻塞 | 自动修复，最多 3 轮 | 每轮汇报 |
| 测试失败 | 🔴 阻塞 | 分析原因，提出修复方案 | 等待确认 |
| Lint Error | 🟡 高 | 自动修复（如可自动） | 批量汇报 |
| 类型错误 | 🔴 阻塞 | 分析并修复 | 每轮汇报 |
| 覆盖率不足 | 🟡 中 | 补充测试建议 | 汇总汇报 |
| Warning | 🟢 低 | 记录，不阻塞 | 可选汇报 |

### 自动修复原则

**可以自动修复的**:
- 格式化问题（Prettier/gofmt）
- 简单的 Lint 问题（未使用变量、导入排序）
- 明显的类型不匹配（有明确修复方案）

**需要用户确认的**:
- 测试失败（可能涉及业务逻辑理解）
- 架构/设计问题
- 性能问题
- 安全警告

## 验证工具链配置

### 项目级配置模板

创建 `package.json` / `pyproject.toml` / `Makefile` 统一入口：

```json
{
  "scripts": {
    "validate": "npm run validate:compile && npm run validate:lint && npm run validate:test",
    "validate:compile": "tsc --noEmit",
    "validate:lint": "eslint . --max-warnings=0",
    "validate:test": "jest --coverage --coverageThreshold=80",
    "validate:fix": "prettier --write . && eslint --fix ."
  }
}
```

```toml
[tool.taskipy.tasks]
validate = "task validate:compile && task validate:lint && task validate:test"
validate_compile = "mypy ."
validate_lint = "ruff check . --select E,W,F,I"
validate_test = "pytest --cov=src --cov-report=term-missing --cov-fail-under=80"
validate_fix = "black . && isort . && ruff check . --fix"
```

```makefile
.PHONY: validate validate-compile validate-lint validate-test validate-fix

validate: validate-compile validate-lint validate-test

validate-compile:
	go build ./...

validate-lint:
	golangci-lint run

validate-test:
	go test -race -coverprofile=coverage.out ./...
	go tool cover -func=coverage.out | grep total | awk '{print $$3}' | sed 's/%//' | awk '{if($$1<80) exit 1}'

validate-fix:
	gofmt -w .
	goimports -w .
```

## 快速验证 vs 完整验证

### 开发中的快速验证（每次保存）

| 检查项 | 工具 | 时间 |
|--------|------|------|
| 编译/类型检查 | tsc/mypy/go build | <5s |
| 当前文件测试 | jest --testPathPattern | <10s |
| 格式化 | prettier/gofmt | <1s |

### 提交前的完整验证

| 检查项 | 工具 | 时间 |
|--------|------|------|
| 全量编译 | 编译器 | 10-30s |
| 全量 Lint | eslint/ruff/golangci-lint | 10-30s |
| 全量单元测试 | jest/pytest/go test | 30s-2m |
| 覆盖率检查 | coverage tools | 包含在上项 |
| 集成测试 | 测试容器/DB | 1-5m |

## 特殊场景处理

### 场景 1: 验证工具未安装

```
检测到验证工具 [tool] 未安装。

选项:
1. 自动安装: [安装命令]
2. 跳过此验证步骤（不推荐）
3. 使用替代工具: [替代方案]

请选择:
```

### 场景 2: 验证超时

```
验证执行超时（>5分钟）。

可能原因:
- 测试死锁
- 无限循环
- 资源竞争

建议:
1. 检查最近的代码变更
2. 单独运行超时测试
3. 增加调试输出

是否继续等待，还是终止检查？
```

### 场景 3:  flaky 测试

```
检测到 flaky 测试: [测试名称]

特征:
- 偶发性失败
- 与代码变更无关
- 重试可能通过

处理方案:
1. 标记为 skip，创建修复任务
2. 增加重试机制（临时方案）
3. 立即修复（如原因明确）

建议选择:
```
