## Phase 1: 需求澄清与确认 (REQUIREMENT)

### 1.1 首次响应模板

当用户提出任务时，你的第一句话必须是：

```
收到任务。在开始任何实现之前，我需要先完成需求澄清。

根据任务描述，我初步理解的需求如下：

【用户原始需求摘要】
- 目标: [一句话概括]
- 范围: [明确包含/不包含的内容]
- 约束: [已知的限制条件]

【待澄清问题】
我有以下问题需要确认：

1. [具体问题1] - 影响[具体方面]
2. [具体问题2] - 影响[具体方面]
...

请回答以上问题，或补充任何我遗漏的重要信息。
```

### 1.2 需求澄清检查清单

必须确认以下维度（如适用）：

| 维度 | 确认内容 |
|------|---------|
| **功能范围** | 必须实现什么？明确排除什么？ |
| **用户场景** | 谁使用？在什么场景下使用？ |
| **输入/输出** | 数据从哪来？输出到哪去？格式是什么？ |
| **性能要求** | 吞吐量？延迟？并发量？ |
| **可靠性** | 可用性目标？故障恢复要求？ |
| **安全** | 认证？授权？数据保护？ |
| **兼容性** | 需要支持哪些版本/平台？ |
| **约束条件** | 技术栈限制？第三方依赖？预算？ |

### 1.3 需求确认标准

只有在以下情况下，才能结束 Phase 1：
- [ ] 所有待澄清问题已得到明确回答
- [ ] 用户确认"需求理解无误，可以进入设计阶段"
- [ ] 已识别并记录所有风险点和假设

## 环境准备 (ENVIRONMENT SETUP)

### 项目启动前必须执行

在开始任何工作前，先检查和准备环境：

```
1. 运行环境检查脚本
   ./scripts/check-env.sh
   
2. 如环境缺失，运行对应语言的 setup 脚本
   ./scripts/setup-python.sh      # Python (uv)
   ./scripts/setup-typescript.sh  # TypeScript (nvm + npm/bun)
   ./scripts/setup-java.sh        # Java (jenv + maven)
   ./scripts/setup-go.sh          # Go (goenv)

3. 验证工具链可用
   make validate  # 或对应语言的验证命令
```

### 推荐工具版本管理

| 语言 | 版本管理器 | 包管理器 | 验证命令 |
|------|-----------|---------|---------|
| Python | `uv` | `uv pip` | `uv --version` |
| TypeScript | `nvm` | `npm`/`bun` | `node --version` |
| Java | `jenv` | `maven` | `java --version` |
| Go | `goenv` | `go mod` | `go version` |

### Docker 沙盒环境（可选但推荐）

提供隔离的、可复现的开发环境：

```bash
# 启动特定语言沙盒
docker-compose -f sandbox/docker-compose.yml up -d python-dev
docker-compose -f sandbox/docker-compose.yml up -d dev  # 通用多语言

# 进入沙盒
docker-compose -f sandbox/docker-compose.yml exec dev bash
```

每个代码变更后必须执行验证：

```

   命令: tsc --noEmit / mypy . / go build / mvn compile
   失败: 立即停止，修复语法错误

Step 2: 静态分析
   命令: eslint . / ruff check . / golangci-lint run
   失败: 修复所有 Error，评估 Warning

Step 3: 单元测试
   命令: jest / pytest / go test / mvn test
   失败: 修复失败的测试，补充缺失的测试
   目标: 覆盖率 ≥80%

Step 4: 代码审查 (Self-Review)
   检查清单:
   - [ ] 符合语言规范
   - [ ] 命名清晰
   - [ ] 有适当注释
   - [ ] 错误处理完善
   - [ ] 无冗余代码
```

## 语言特定规范

### Java
- 遵循 Google Java Style Guide
- 使用 final 修饰不可变变量和参数
- 优先使用 Optional 而非 null
- 异常处理: 受检异常用于可恢复错误，运行时异常用于编程错误
- 依赖注入: 使用 Spring/Guice 等框架

### Python
- 遵循 PEP 8 + Google Python Style Guide
- 类型注解: 所有函数必须有类型提示
- 使用 dataclasses 或 Pydantic 定义数据结构
- 异常: 自定义异常继承自特定基类
- 依赖管理: 使用 pyproject.toml + poetry/uv

### TypeScript
- 遵循 Google TypeScript Style Guide
- 严格模式: strict: true
- 禁止 any，使用 unknown + 类型守卫
- 优先使用 interface 定义对象形状
- 错误处理: 使用 Result/Either 模式或明确抛出

### Go
- 遵循 Effective Go + Uber Go Style Guide
- 错误处理: 显式检查，错误信息小写开头
- 接口: 小接口优于大接口
- 并发: 使用 context 传递取消信号
- 测试: 表格驱动测试，t.Parallel()

## 通用代码规范

### 命名
- 变量/函数: 清晰表达意图，避免缩写（除非行业标准）
- 类/模块: 名词，单一职责
- 布尔值: 使用 is/has/should/can 前缀

### 函数设计
- 单一职责: 一个函数只做一件事
- 参数数量: ≤3 个，超过使用配置对象/结构体
- 返回值: 错误作为最后一个返回值（Go风格）或用异常/Result
- 副作用: 在函数名中明示（如 `saveXxx`, `mutateXxx`）

### 注释
- 为什么 > 做什么 > 怎么做
- 公共 API 必须有文档注释
- 复杂算法必须解释原理

### 错误处理
- 错误信息必须包含上下文（哪个操作、哪个实体）
- 区分用户错误（4xx）和系统错误（5xx）
- 日志记录: 错误发生点记录，传播时包装

## 禁止事项

- ❌ 直接开始写代码而不经过需求确认
- ❌ 设计未确认就进入实现
- ❌ 一次提交超过 200 行变更（不含测试）
- ❌ 使用 "TODO" 或 "FIXME" 而不创建跟踪任务
- ❌ 忽略编译器/静态分析警告
- ❌ 提交未测试的代码
- ❌ 在代码中硬编码配置
- ❌ 不更新 PLAN.md 状态就继续工作
- ❌ 跳过验证步骤直接提交代码
